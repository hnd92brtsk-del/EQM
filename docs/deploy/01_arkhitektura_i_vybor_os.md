# Архитектура развёртывания и выбор ОС

## Рекомендуемая схема

Для проекта EQM рекомендуется схема из **двух серверов**:

1. `eqm-app-01` (`192.168.10.20`)  
   Назначение: `Nginx + frontend + backend + PostgreSQL`

2. `eqm-llm-01` (`192.168.10.21`)  
   Назначение: `LM Studio + модель phi3-mini-4k-instruct`

Такой вариант оптимален для вашей нагрузки `10-20` одновременных пользователей, потому что:

- основной бизнес-контур EQM не зависит от вычислительной нагрузки LLM;
- при проблемах с моделью основное приложение продолжает работать;
- администрировать `Docker + Nginx + PostgreSQL` проще на отдельном app-сервере;
- GUI для LM Studio удобнее держать на отдельной машине.

## Почему Linux, а не Windows

### Почему для app-сервера лучше Linux

Для `eqm-app-01` лучше использовать **Ubuntu Server 24.04 LTS**.

Причины:

- Docker и Docker Compose на Linux проще и стабильнее в сопровождении;
- reverse proxy `Nginx` штатно используется именно в Linux production-сценариях;
- меньше расход ресурсов, чем у Windows Server;
- проще резервное копирование, ротация логов, работа с `systemd`, `tar`, `rsync`;
- проще переносить образы и поднимать контейнеры в полностью офлайн-среде.

### Почему для LLM-сервера нужен другой дистрибутив

Для `eqm-llm-01` лучше использовать **Ubuntu Desktop 22.04.5 LTS**.

Причины:

- LM Studio на Linux работает как `AppImage`;
- нужен GUI для первичной настройки модели, проверки загрузки, тестирования API и памяти;
- серия Ubuntu `22.04 LTS` сейчас более безопасный выбор именно под LM Studio, чем более новые desktop-сборки.

## Почему не рекомендуется Windows Server как основной вариант

Windows Server здесь не запрещён, но он менее практичен:

- сложнее production-эксплуатация Docker-контейнеров для такого стека;
- хуже типовой опыт по `Nginx + FastAPI + PostgreSQL` в on-prem Linux-среде;
- больше ручной настройки;
- сложнее повторяемое сопровождение для новичка-сисадмина.

Вывод: **основной production-вариант нужно строить на Linux**.

## Почему Nginx без Apache

Для этого проекта не нужен `Apache` и не нужна связка `nginx + apache`.

Причины:

- frontend собирается в статический bundle;
- backend — обычный HTTP API на FastAPI;
- PHP и `.htaccess` в проекте отсутствуют;
- WebSocket сейчас не используется;
- `Nginx` проще как единая точка входа: он отдаёт frontend и проксирует `/api/...` в backend.

Итог:

- **reverse proxy**: `Nginx`
- **backend runtime**: `uvicorn`
- **frontend delivery**: статический build через `Nginx`
- **БД**: `PostgreSQL 16`
- **LLM**: `LM Studio` на отдельном сервере

## Когда действительно понадобится третий сервер

На старте третий сервер не нужен. Он понадобится только в одном из случаев:

- количество пользователей заметно вырастет выше текущего диапазона;
- появится тяжёлая аналитика или интеграции;
- понадобится отдельный сервер под PostgreSQL;
- понадобятся требования по отказоустойчивости;
- `phi3` будет заменена на более тяжёлую модель и начнёт требовать отдельный GPU-хост.

## Финальная рекомендация

Используйте именно эту схему:

- `eqm-app-01`: **Ubuntu Server 24.04 LTS**
- `eqm-llm-01`: **Ubuntu Desktop 22.04.5 LTS**
- приложение EQM: **Docker Compose**
- входной веб-сервер: **Nginx**
- LLM: **LM Studio GUI**

Это самый практичный вариант для вашей задачи на локальном сервере без интернета.
